## 事業議論の視点

迷ったとき、AIの提案を評価するときは、以下のレンズで見る。

### マクロの視点を持つ（具体と抽象の行き来）

機能の追加・削除・修正を行うときは、ミクロの改善視点だけではなく、
システム全体及び事業全体への影響を鑑みるマクロの視点を必ず持つこと。

判断ステップ：
1. ミクロ（その機能単体）で見て、改善余地はあるか
2. マクロ（システム全体・事業全体）で見て、その変更は何に影響するか
3. 具体（個別の事象）と抽象（全体パターン）を往復して判断する

### 「削除推奨」の調査結果は事業影響を必ず再確認する

AIが「使われてない」「動いてない」「フラグオフ状態」と検出した機能は、
技術的には削除候補に見えるが、それが「事業的に不要」とは限らない。

判断ステップ：
1. AIから「削除推奨」が出たら、まず事業影響を確認する
2. 「動いていない」のか「動かしてないだけ」なのかを区別する
3. 改善の余地がある機能は、削除ではなく改善・統合を検討する
4. 削除前に「この機能を完全に捨てて事業上問題ないか」を必ず自問する

具体例（2026-05-08 の学び）：
- list-tool の Phase 5.2（voice-report / daido-kenco の検索クエリ逆引き）
- Claude Code が「3ヶ月 0件継続、デッドコード」として削除推奨
- 実態は「機能は正しい設計、クエリ調整不足で 0件」
- 媒体は健康経営優良法人を多数掲載しており事業上重要
- 削除ではなく E1 で「クエリ拡充 + 統合」する判断に変更
